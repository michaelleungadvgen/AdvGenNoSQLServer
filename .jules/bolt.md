## 2026-10-23 - Avoid intermediate list allocations for deferred filtering pipelines
**Learning:** Returning `IEnumerable<T>` and immediately converting it to `.ToList()` inside a basic storage class (e.g. `InMemoryDocumentCollection.GetAll()`) represents a massive performance penalty when the result is directly fed into a `Where` filter anyway (`FilterEngine.Filter(documents).ToList()`). In .NET, `ConcurrentDictionary.Values` acquires locks and allocates an array to provide a moment-in-time snapshot. Iterating the dictionary directly (`foreach (var kvp in _documents) yield return kvp.Value;`) allows for truly zero-allocation lazy evaluation.
**Action:** Next time you see `.ToList()` or explicit list instantiations with `.Add()` in a data retrieval layer, look at the caller. If the caller enumerates it for filtering or projection, switch the data retrieval layer to return the lazy enumerable or an iterator block (`yield return`) to prevent O(N) memory overhead and garbage collection pressure. Additionally, avoid `.Values` on `ConcurrentDictionary` if true zero-allocation enumeration is desired.

## 2026-10-23 - Avoid replacing safe materialized snapshots with dangerous lazy iterators across component boundaries
**Learning:** Returning `IEnumerable<T>` using `yield return` from a dictionary iterator (e.g. `foreach (var kvp in _tombstones) yield return kvp.Value;`) instead of `.ToList()` inside a basic storage class creates an extremely dangerous lazy iteration semantic. If the caller iterates over this returned `IEnumerable` and concurrently removes items from the collection (a common pattern for a GarbageCollector), it will throw an `InvalidOperationException` ("Collection was modified; enumeration operation may not execute") or introduce subtle visibility bugs if it's a `ConcurrentDictionary`.
**Action:** Before converting `.Values.ToList()` to a lazy `yield return` iterator, you MUST trace how the caller consumes the data. If the caller modifies the underlying state during enumeration, the `.ToList()` snapshot is required for safety and is an intentional design choice, not an optimization candidate.

## 2026-10-23 - Consolidate multiple O(N) linear scans on collections
**Learning:** Using multiple LINQ aggregation methods (like `.Sum()`, `.Average()`, `.Min()`, `.Max()`) in sequence on a collection (especially `ConcurrentDictionary.Values`) creates multiple `O(N)` linear scans. Eagerly calling `.ToList()` before aggregation to avoid lock contention still forces a massive O(N) heap allocation.
**Action:** When you see sequential LINQ aggregation methods being performed against the same collection to populate statistics or check rebalance thresholds, consolidate them into a single O(N) `foreach` loop that tracks all variables concurrently. This avoids redundant traversal overhead, eliminates multiple lock acquisitions on concurrent collections, and bypasses the need for large intermediary `.ToList()` arrays.
